Голосовой AI-агент: задаёт вопросы по цели интервью, слушает голосовые ответы, распознаёт их через Whisper и собирает структурированный чеклист созвона с клиентом.

i Коротко: что это и как работает

Пользователь нажимает «Начать», агент задаёт 3 вопроса. Человек отвечает голосом — запись уходит на сервер, где Whisper превращает речь в текст, а модель minimax-m3 анализирует ответы и задаёт следующие уточняющие вопросы. Так 3 раунда (до 9 вопросов), затем агент формирует чеклист в Markdown, который можно скачать.

1 Что просило задание — и что сделано

Курс «Разработка с AI: автоматизация и продуктивность». Три пункта задания:

Пункт заданияКак реализовано
1. Базовый голосовой агент с целью; на основе цели и контекста задаёт вопросы, отвечаемые голосом. Агент имеет цель «заполнить чеклист созвона». На её основе и на основе предыдущих ответов (контекст) он генерирует вопросы. Ответы — только голосом (Whisper). готово
2. Опубликовать веб-сервис на Hugging Face и Vercel (или видео). Бэкенд (FastAPI + Whisper) — на Hugging Face Space. Фронтенд (Next.js) — на Vercel. Оба задеплоены и работают публично. готово
3. Дать агенту более сложную обработку / поэкспериментировать с моделями. Адаптивные уточняющие вопросы (раунды 2–3 строятся на анализе предыдущих ответов) + классификация каждого пункта чеклиста по статусам (confirmed / needs_clarification / not_discussed). См. раздел «Логика агента». готово

2 Архитектура

Два независимых сервиса. Браузер — это «руки и уши», вся логика — на бэкенде.

🖥️ FRONTEND — Vercel
Next.js 15 · React 19 · TypeScript · Tailwind · shadcn-style UI
Запись с микрофона (MediaRecorder) Экраны: старт → интервью → результат Markdown-превью + скачивание .md
↓ ↑ HTTPS · multipart/form-data (webm-аудио) · JSON
🧠 BACKEND — Hugging Face Space (Docker)
FastAPI · Python 3.11 · ffmpeg
Router /api/session/* Session Manager (in-memory) Agent (раунды, чеклист) Markdown Generator
🎧 Whisper-small (локально, transformers+torch, CPU)
↓ ↑ HTTPS · OpenAI-совместимый API
☁️ OpenRouter — LLM
модель minimax/minimax-m3 — генерация вопросов, анализ, чеклист
Почему так: распознавание речи (Whisper) — тяжёлое и приватное, ему место на сервере рядом с агентом. Браузер только пишет звук и показывает UI. Это ровно разделение из исходного архитектурного документа: Frontend (Vercel) ⟷ Backend (HF) ⟷ внешний LLM.

3 Поток данных (один раунд)

  1. Старт сессииБраузер → POST /api/session/start. Бэкенд просит у minimax-m3 первые 3 вводных вопроса и возвращает их + session_id.
  2. Запись голосаНа каждый вопрос пользователь жмёт «Записать». MediaRecorder создаёт аудио webm/opus прямо в браузере.
  3. Отправка ответовВсе 3 аудио уходят одним запросом → POST /api/session/{id}/submit (multipart/form-data).
  4. Распознавание (STT)Бэкенд для каждого аудио: webm → ffmpeg → wav 16kHz mono → Whisper-small → текст.
  5. Анализ + следующие вопросыТранскрипты + цель интервью → minimax-m3. Модель пишет короткое резюме и уточняющие вопросы следующего раунда.
  6. Финал (после раунда 3)Все ответы → minimax-m3 → структурированный чеклист (категории + статусы). Бэкенд рендерит Markdown и отдаёт его.
  7. РезультатФронтенд показывает чеклист с чекбоксами и даёт кнопку «Скачать .md».

4 Из чего собрано (компоненты)

Frontend · Vercel

Next.js 15 + React 19

Экраны Landing → Interview → Results как состояние одной страницы. Хук useAudioRecorder управляет микрофоном. Tailwind + shadcn-style компоненты (Button, Card, Progress).

Backend · Hugging Face

FastAPI

Эндпоинты /start, /transcribe, /submit, /results, /download, /health. Сессии — in-memory словарь. Запакован в Docker, порт 7860.

STT · локально

openai/whisper-small

HuggingFace transformers pipeline на CPU. Модель скачивается на этапе сборки Docker — быстрый холодный старт. ffmpeg конвертирует браузерный webm в wav.

LLM · OpenRouter

minimax/minimax-m3

Через OpenAI-совместимый SDK (base_url=openrouter.ai). Три функции агента: вопросы раунда, анализ, итоговый чеклист. Ответы — строгий JSON, парсятся устойчиво.

5 Логика агента (почему это «агент с целью»)

У агента есть зафиксированная цель:

# backend/app/prompts.py
GOAL = "Провести голосовое интервью с клиентом и заполнить чеклист созвона:
        название и суть проекта, цели и метрики, сроки, бюджет,
        технические требования и интеграции, риски и открытые вопросы."

Каждый запрос к модели = цель + контекст (все предыдущие ответы). Поэтому вопросы раунда 2–3 не шаблонные, а зависят от того, что человек уже сказал. Пример из реального прогона:

Раунд 1 — вводный

«Как называется проект и в чём суть?» · «Какую бизнес-задачу решаете?» · «Сроки и бюджет?»

Раунд 2 — адаптивный

Модель увидела «доставка еды, бюджет не назван» → спросила про целевую аудиторию, конкретные метрики и вилку бюджета.

Финал: каждый пункт чеклиста получает статус, что и есть «более сложная обработка» (пункт 3 задания):

confirmed — факт чётко получен   needs_clarification — затронуто, но мутно   not_discussed — важная тема не раскрыта

6 Компромиссы и отклонения от плана

Честно: где реализация отличается от исходного архитектурного документа и почему. (Этот раздел просили включить отдельно.)

В документе былоСделаноПочему
LLM = Claude API (Anthropic, Haiku) LLM = OpenRouter, minimax/minimax-m3 Прямое требование. Ключ Anthropic не используется; всё через OpenRouter.
Агент на LangGraph (граф узлов) Агент на чистом Python (простая машина состояний по раундам) Поток линейный (раунд 1→2→3). LangGraph здесь — лишние тяжёлые зависимости и точки отказа. Шаги те же, код прозрачнее и легче отлаживать.
Роуты /session/[id] и /results/[id] Одна страница со сменой экранов (state machine) Аудио-blob'ы живут в памяти браузера; при переходах между роутами их легко потерять. Один stateful-экран надёжнее. Стек (Next.js 15 + Tailwind + shadcn-style) — как в плане.
Markdown генерирует LLM Markdown рендерит Python из структурированного JSON Детерминированный формат: чекбоксы и категории всегда корректны, не зависят от «настроения» модели.
Next.js 15.1 Next.js 15.5.19 Vercel блокирует деплой версий с известными CVE. Обновил до пропатченной 15.x.
Деплой через браузер-логин (скриншот) Деплой через write-токен HF и токен Vercel (headless) В headless-среде браузер-OAuth неудобен. Токен — эквивалентно и автоматизируемо.
ALLOWED_ORIGINS = домен Vercel CORS = * упрощение Чтобы не привязываться к домену. Легко ужесточить: задать секрет ALLOWED_ORIGINS у Space.
Хранилище сессий (MVP — in-memory) In-memory словарь (как в MVP плана) Сессии сбрасываются при перезапуске Space. Для учебного проекта достаточно; для prod — Redis/SQLite.
Примечание про модель: в скриншоте minimax/minimax-m3 на OpenRouter оказалась реальной моделью (≈ $0.30 / $1.20 за 1M токенов вход/выход) — её и используем, как и просили.

7 Как запустить локально

Backend (FastAPI + Whisper)

cd backend
bash setup_local.sh        # venv + torch(CPU) + deps
# создать .env из .env.example c ключом OpenRouter
source .venv/bin/activate
uvicorn app.main:app --port 7860

Frontend (Next.js)

cd frontend
npm install
# .env.local: NEXT_PUBLIC_API_URL=http://localhost:7860
npm run dev   # http://localhost:3000
Микрофон в браузере работает только на https:// или http://localhost — это требование безопасности браузеров, не баг.

8 Безопасность ключей

Важно. Ключи (OpenRouter, Vercel-токен, HF-токен) во время работы передавались в переписке — их нужно считать скомпрометированными и пересоздать:
  • OpenRouter → openrouter.ai/settings/keys (revoke + new), затем обновить секрет у HF Space.
  • Vercel → vercel.com/account/tokens (delete + new).
  • Hugging Face → huggingface.co/settings/tokens (revoke + new).
В коде/репозитории ключей нет: на бэкенде ключ — это Secret у HF Space, локально — в .env (в .gitignore).
